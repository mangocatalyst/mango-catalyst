## Important

1. **Light-band amber rule contradiction.** `src/app/globals.css:24-26` locks amber as "BUTTON FILL ONLY" on light bands (links/accents must be navy), but `src/components/ui/SectionHeading.tsx:21`, `src/components/ui/Card.tsx:29` (the `accent` prop), and `src/app/services/page.tsx:184` all render amber ticks/accents inside light-tone sections. Fix: swap those instances to navy, or revise the documented rule to explicitly allow small decorative (aria-hidden) amber marks and reconcile the comment.

2. **Unfinished content live on the homepage.** `src/components/sections/CredibilityBar.tsx:39-44` renders a dashed "Photo coming soon" placeholder in the proof section of the live site, confirmed present in the deployed HTML. Fix: ship the approved headshot or remove the placeholder box until it's ready.

3. **MN-ITS tutorial is entirely placeholder content in production.** `public/mn-its/` has zero files, so every asset referenced by `src/components/mnits/DemoMedia.tsx:7-13` and rendered in `src/app/mn-its/page.tsx:134-182` (one video, five screenshots) falls back to "Demo video coming shortly" / "Screenshot coming" boxes, confirmed live. Fix: add the approved masked media, or hide/collapse the tutorial section until assets ship.

4. **Contact email is a dead-end and split across three sources of truth.** `src/lib/constants.ts:23` leaves `SITE.email` blank, so `src/app/contact/page.tsx:239-248` never renders an email line — yet the failed-submission message at `src/app/contact/page.tsx:143-150` tells the visitor to "email me directly" with no address shown anywhere on that page. Meanwhile `src/app/privacy/page.tsx:140-141` hardcodes `hello@mangocatalyst.com` outside the `SITE` constant that every other page/schema field reads from. Fix: set the approved address once in `SITE.email` and let contact, privacy, footer, CTA fallback, JSON-LD, and the delivery fallback all read from it — or remove the "email me directly" instruction if no address is approved yet.

5. **Contact API has no rate limiting or input bounds.** `src/app/api/contact/route.ts` silently drops only honeypot hits; the fast-fill check is advisory (flags, never blocks), and there is no per-IP/time rate limit and no length cap on `name`/`email`/`business`/`message`. Any request that clears the honeypot is unlimited, so a script can exhaust sending quota, create cost, or flood the inbox, and an oversized paste is relayed straight into the outbound email. Fix: add provider/edge-level rate limiting (e.g. Vercel Firewall or an edge-config token bucket — an in-memory counter won't work across serverless instances) plus server-side length caps on all four fields.
